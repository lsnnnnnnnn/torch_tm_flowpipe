# Technical validation and usefulness evidence

This document separates three claims that are easy to conflate during a report.

## 1. What was implemented

The core library implements a fixed-step, fixed-order Taylor-model flowpipe
kernel for polynomial plant ODEs.  The key multi-step modes are:

- `range_only`: after each step, compress the endpoint Taylor model to an
  interval box and restart from identity Taylor models over that new box.
- `dependency_preserving`: after each step, substitute the local time variable
  `tau = h`, drop the inactive local time variable, and pass the endpoint Taylor
  model directly into the next step.

The second mode preserves symbolic dependency on the original initial-state
variables across steps.  It is useful when box compression is the dominant source
of wrapping/error growth.

## 2. What the tests prove

The automated tests check implementation-level invariants:

- interval and polynomial arithmetic smoke tests;
- Taylor-model substitution/drop-variable behavior;
- one-step and multi-step flowpipe validation status;
- dependency-preserving endpoints no longer contain the local time variable;
- sampled exact/RK4 trajectories fall inside the final box for bundled small
  benchmarks;
- Flow* exporter/parser/runner plumbing, including skipped Flow* runs that still
  export the generated C++ case.

These tests are regression tests.  They are not a production-level floating-point
soundness proof.

## 3. How usefulness is measured

Usefulness is measured by comparing the final reachable-box width of
`dependency_preserving` against `range_only` at the same system, step size,
horizon, and Taylor order:

```text
ratio = final_width_sum(dependency_preserving) / final_width_sum(range_only)
```

Interpretation:

- `ratio < 1`: dependency-preserving is tighter and helped.
- `ratio ≈ 1`: no practical difference on that case.
- `ratio > 1`: dependency-preserving regressed, usually because polynomial term
  growth and endpoint remainder accumulation outweighed saved dependency.

The comparison summary is generated by:

```bash
python comparisons/flowstar/summarize_comparison.py \
  outputs/flowstar_comparison.csv \
  --out outputs/flowstar_comparison_summary.md
```

The Flow* comparison is separate.  It only becomes a real numerical comparison
when Flow* is installed and the generated C++ cases compile/run.  If Flow* is not
available, the CSV should show `tool=flowstar,status=skipped`; no torch-vs-Flow*
width claim should be made from those rows.

## 4. Correct Flow* comparison architecture

The current `chenxin415/flowstar` repository is a C++ toolbox/static-library
interface.  Therefore this repo compares against Flow* by generating C++ cases,
compiling them against `flowstar-toolbox/libflowstar.a`, executing the resulting
binary, parsing Flow* plotting/range output, and writing box metrics into the
same CSV as the PyTorch Taylor-model results.

This is intentionally an external comparison harness.  It is not a Flow* binding
inside `src/torch_tm_flowpipe`, and it does not modify the core algorithm.

## 5. Limits to report clearly

Do not claim this project has replaced Flow* or CROWN-Reach.  The current scope
is:

```text
plant-only polynomial ODEs;
fixed step size;
fixed Taylor order;
box-enclosure comparison;
research prototype floating-point treatment.
```

Not supported yet:

```text
CROWN/auto_LiRPA controller integration;
Jacobian/sensitivity bounds;
non-polynomial sin/cos/exp/log in the torch core;
hybrid automata guards/jumps;
adaptive step/order;
symbolic remainder;
branch-and-bound refinement;
production-level sound verifier claims.
```
